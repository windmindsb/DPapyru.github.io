## 如何贡献

### 内容贡献

我们特别欢迎以下类型的内容贡献：

1. **新教程**：涵盖泰拉瑞亚Mod开发的各个方面
2. **代码示例**：实用的代码片段和完整示例
3. **概念解释**：对复杂概念的清晰解释
4. **最佳实践**：开发经验和技巧分享
5. **翻译工作**：将现有内容翻译为其他语言

#### 内容贡献流程

1. **确认内容范围**：
   - 查看现有文档，避免重复
   - 在Issues中讨论您计划创建的内容
   - 确认内容符合项目目标

2. **创建内容**：
   - 在`docs/`目录下创建新的Markdown文件
   - 遵循[写作指南](#写作指南)
   - 确保内容准确、清晰、有用

3. **提交内容**：
   - 创建Pull Request
   - 在PR描述中说明内容的用途和目标受众
   - 等待维护者审核

### 代码贡献

我们欢迎以下类型的代码贡献：

1. **功能改进**：增强网站功能
2. **错误修复**：修复现有问题
3. **性能优化**：提高网站性能
4. **代码重构**：改善代码结构
5. **测试增强**：提高测试覆盖率

#### 代码贡献流程

1. **设置开发环境**：
   ```bash
   # 克隆仓库
   git clone https://github.com/DPapyru/DPapyru.github.io.git
   cd DPapyru.github.io
   
   # 创建功能分支
   git checkout -b feature/your-feature-name
   ```

2. **进行开发**：
   - 遵循[代码风格](#代码风格)
   - 确保代码通过测试
   - 添加必要的文档

3. **提交代码**：
   - 遵循[提交规范](#提交规范)
   - 创建Pull Request
   - 在PR描述中详细说明更改内容

### 问题报告

如果您发现了问题或有改进建议，请通过GitHub Issues报告：

1. **搜索现有Issues**：避免重复报告
2. **使用适当的模板**：选择最匹配的问题模板
3. **提供详细信息**：包括复现步骤、预期行为和实际行为
4. **添加相关标签**：帮助维护者分类和处理

## 写作指南

### Markdown规范

我们使用标准Markdown语法，并遵循以下规范：

1. **文件命名**：
   - 使用小写字母和连字符：`file-name.md`
   - 避免使用空格和特殊字符
   - 保持名称简洁且描述性强

2. **标题结构**：
   ```markdown
   # 主标题（H1）
   
   ## 二级标题（H2）
   
   ### 三级标题（H3）
   
   #### 四级标题（H4）
   ```
   - 每个文档应只有一个H1标题
   - 按层级顺序使用标题，不要跳级
   - 标题使用简洁、描述性的文本

3. **列表**：
   ```markdown
   - 无序列表项1
   - 无序列表项2
     - 嵌套项
     - 另一个嵌套项
   
   1. 有序列表项1
   2. 有序列表项2
   ```

4. **链接**：
   ```markdown
   [链接文本](URL)
   
   [内部链接](file-name.md)
   
   [带标题的链接](URL "链接标题")
   ```

5. **图片**：
   ```markdown
   ![替代文本](图片URL)
   
   ![带标题的图片](图片URL "图片标题")
   ```

### 代码示例

1. **行内代码**：使用反引号 ` ` 包裹
   ```markdown
   使用 `Mod` 类来创建你的Mod
   ```

2. **代码块**：使用三个反引号，指定语言
   ```markdown
   ```csharp
   using Terraria;
   using Terraria.ModLoader;
   
   public class MyMod : Mod
   {
       // 代码内容
   }
   ```
   ```

3. **代码注释**：
   - 为复杂代码添加解释
   - 使用注释说明代码目的
   - 避免显而易见的注释

4. **代码格式**：
   - 使用一致的缩进（4个空格）
   - 遵循C#编码规范
   - 确保代码可编译和运行

### 图片和媒体

1. **图片格式**：
   - 优先使用PNG格式（截图）
   - 使用SVG格式（图标和简单图形）
   - 避免使用过大的图片文件

2. **图片大小**：
   - 宽度不超过800像素
   - 文件大小不超过200KB
   - 使用适当的压缩

3. **替代文本**：
   - 为所有图片提供描述性替代文本
   - 替代文本应简洁且准确

## 代码风格

### JavaScript代码风格

1. **命名约定**：
   - 变量和函数：驼峰命名法 `camelCase`
   - 常量：大写下划线 `UPPER_SNAKE_CASE`
   - 类和构造函数：帕斯卡命名法 `PascalCase`

2. **代码格式**：
   - 使用2个空格缩进
   - 使用单引号
   - 每行不超过80个字符

3. **注释**：
   ```javascript
   // 单行注释
   /**
    * 多行注释
    * @param {参数类型} 参数名 - 参数描述
    * @returns {返回类型} 返回值描述
    */
   ```

### CSS代码风格

1. **命名约定**：
   - 使用连字符分隔：`class-name`
   - 使用有意义的名称
   - 避免使用缩写

2. **代码组织**：
   - 按功能分组样式
   - 使用注释分隔不同部分
   - 保持一致的顺序

## 提交规范

我们使用[约定式提交](https://www.conventionalcommits.org/zh-hans/)规范：

### 提交消息格式

```
<类型>[可选的作用域]: <描述>

[可选的正文]

[可选的脚注]
```

### 类型说明

- `feat`: 新功能
- `fix`: 错误修复
- `docs`: 文档更改
- `style`: 代码格式化（不影响功能）
- `refactor`: 代码重构
- `test`: 添加或修改测试
- `chore`: 构建过程或辅助工具的变动

### 示例

```bash
git commit -m "feat(search): 添加搜索功能"

git commit -m "fix(navigation): 修复移动端导航菜单问题"

git commit -m "docs: 更新README安装说明"
```

## 审核流程

### Pull Request审核

1. **自动检查**：
   - 代码风格检查
   - 测试通过
   - 构建成功

2. **人工审核**：
   - 代码质量和设计
   - 功能正确性
   - 文档完整性

3. **审核标准**：
   - 代码清晰可读
   - 功能正确实现
   - 符合项目规范
   - 有适当的测试

### 内容审核

1. **内容质量**：
   - 准确性和时效性
   - 清晰度和完整性
   - 目标受众适合性

2. **格式检查**：
   - Markdown语法正确
   - 链接有效
   - 图片显示正常
